# 就业模块与居民模块统一化改进计划

## 1. 存在的不统一问题

### 1.1 列表页实现差异
- **就业列表页**：直接使用`List<Employment>`接收数据，未使用`EmploymentResponse`分页响应对象
- **居民列表页**：使用`ResidentResponse`分页响应对象，结构更规范
- **跳转extra key**：就业列表页使用字符串"employment_id"，而居民列表页使用常量`EXTRA_RESIDENT_ID`

### 1.2 详情页实现差异
- **日期显示**：就业详情页直接调用`toString()`显示日期，未格式化
- **就业状态**：直接显示"在职"，未根据`endDate`是否为null动态判断

### 1.3 常量使用不统一
- 就业模块虽然定义了`EXTRA_EMPLOYMENT_ID`常量，但列表页未使用

## 2. 统一化改进方案

### 2.1 列表页改进 (`EmploymentListActivity.java`)
- 修改数据获取逻辑，使用`EmploymentResponse`接收分页数据
- 使用`EXTRA_EMPLOYMENT_ID`常量替代字符串"employment_id"
- 统一分页数据处理逻辑，参考`ResidentListActivity`

### 2.2 详情页改进 (`EmploymentActivity.java`)
- 添加日期格式化显示，使用`SimpleDateFormat`
- 添加就业状态动态判断逻辑：
  - 若`endDate`为null，显示"在职"
  - 若`endDate`不为null，显示"离职"

### 2.3 确保常量统一使用
- 在所有相关页面统一使用常量作为Intent extra key

## 3. 改进文件清单

1. **`EmploymentListActivity.java`**
   - 修改`loadEmployments()`方法，使用`EmploymentResponse`
   - 修改`loadMoreEmployments()`方法，使用`EmploymentResponse`
   - 使用常量`EXTRA_EMPLOYMENT_ID`替代字符串

2. **`EmploymentActivity.java`**
   - 修改`displayEmploymentDetails()`方法，添加日期格式化
   - 添加就业状态判断逻辑

3. **相关响应类检查**
   - 确保`EmploymentResponse`结构与`ResidentResponse`一致

## 4. 预期效果

- 两个模块的代码结构和实现逻辑完全统一
- 提高代码可维护性和一致性
- 减少后续开发中的困惑和错误
- 统一的用户体验
